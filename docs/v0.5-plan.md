# netwatch-cli V0.5 Plan

## 背景

macOS 的 `arp -a` 能返回大量局域网设备的 MAC 地址，但 V0.3 的解析逻辑过于严格：

- 只接受每段两位的 MAC。
- 只识别 `<incomplete>`，没有识别 `(incomplete)`。
- 没有规范化 `aa:39:b3:b:8b:a` 这类一位 hex 段。

这导致扫描结果中很多 MAC 显示为 `-`。

## 修复点

- 重写 `scanner.py` 中的 `parse_arp_output()`。
- 支持 macOS `arp -a` 格式：
  - `? (192.168.31.75) at a0:9a:8e:84:82:f6 on en0 ifscope permanent [ethernet]`
  - `? (192.168.31.83) at 8c:de:f9:e2:a2:c4 on en0 ifscope [ethernet]`
  - `hostname.local (192.168.31.83) at 8c:de:f9:e2:a2:c4 on en0 ifscope [ethernet]`
- 跳过 incomplete 记录：
  - `? (192.168.31.60) at (incomplete) on en0 ifscope [ethernet]`
- 新增 `normalize_mac_address()`：
  - 支持一位或两位 hex 段。
  - 输出统一为六段两位小写格式。
  - 示例：`aa:39:b3:b:8b:a` -> `aa:39:b3:0b:8b:0a`
- macOS 下优先执行 `arp -an` 获取 numeric ARP 表，避免 `arp -a` 因反向解析耗时过长而超时，导致整张 ARP 表被丢弃。
- `arp -a` 仅作为 hostname enrichment 的 best-effort 步骤；即使失败，也不会影响 MAC 地址补全。
- 保持扫描逻辑在 ping 扫描结束后重新读取 ARP 表，再按 IP 补全 MAC。

## 测试覆盖

新增 `tests/test_scanner.py`，覆盖：

- 普通 MAC。
- `permanent` MAC。
- 一位 hex 段 MAC。
- `(incomplete)` 记录。
- `hostname.local` 格式。
- `arp -an` numeric 输出兜底。
- 扫描结束后再读取 ARP 表补全 MAC/hostname。

## 限制

MAC 地址来自本机 ARP cache，因此通常要求目标设备和本机处在同一个二层局域网内。跨 VLAN、跨子网、隔着路由器或 ARP cache 不完整时，MAC 仍可能显示为 `-`。
