# netwatch-cli V0.8.3 Plan

## 问题背景

V0.8.2 已经接入 official Ookla CLI，并支持 `--interface en0` 避开 `utun` / TUN 出口。但实际环境中，Ookla 自动选服仍经常落到 Hong Kong / Tokyo，下载结果只有数 Mbps；而 `speedtest.cn` 网页可以选择“广东移动_Vixtel_1”，Ping 约 7ms，接近千兆。

这说明当前瓶颈主要是 Ookla server discovery 与广州移动线路不匹配，而不是本机网卡或宽带一定存在问题。

## 实现目标

- 增加测速质量诊断，不再盲信任何一次测速结果。
- 支持保存常用 Ookla server id 到 `~/.netwatch/config.json`。
- 普通“带宽测速”优先询问是否使用已保存服务器。
- preferred server 失败时自动回退到自动测速。
- 高级功能支持保存最近一次成功测速服务器、清除配置、查看配置、查看最近一次 raw 摘要。
- 高级功能提供 `speedtest.cn` 网页对照入口，但不调用私有 API。
- 文档明确说明 Ookla CLI 与 `speedtest.cn` 服务器网络不同。

## 配置设计

配置文件：

```text
~/.netwatch/config.json
```

结构：

```json
{
  "preferred_speedtest": {
    "backend": "official-ookla-cli",
    "server_id": "xxxxx",
    "server_name": "xxx",
    "location": "xxx",
    "interface": "en0"
  }
}
```

只保存非敏感测速偏好，不保存密码、stok、token 或公网 IP。

## 质量诊断规则

满足任一条件就提示当前结果可能不代表真实最大带宽：

- `ping > 80ms`
- `jitter > 30ms`
- `packet_loss > 1%`
- `download_mbps < 50`

提示原因包括服务器距离远、Ookla 自动选服不适合当前运营商、代理/TUN/VPN 影响，以及建议使用指定 server id 或浏览器测速对照。

## 后端降级策略

1. 如果用户选择保存的 preferred server，优先用 `server_id + interface` 测速。
2. 如果 preferred server 失败，回退到自动测速。
3. 自动测速仍按优先级：
   - official Ookla CLI
   - LibreSpeed CLI
   - Python speedtest-cli
4. Python fallback 继续保留，但明确提示结果可能偏低。

## speedtest.cn 边界

- `speedtest.cn` 网页可以作为用户环境的对照基准。
- Ookla CLI 与 `speedtest.cn` 使用的服务器网络不同，不保证能找到“广东移动_Vixtel_1”。
- 当前版本不逆向 `speedtest.cn` 私有网页 API。
- 未来如果有正式 SDK/API 授权，可以作为独立后端接入。
